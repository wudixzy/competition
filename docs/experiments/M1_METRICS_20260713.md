# M1 Streaming Metrics - 2026-07-13

## Scope

This experiment implements `E-METRIC-01` and `E-METRIC-02` without changing
the model, runtime, scheduler, or fixed evaluator command.

The previous benchmark treated `completion_tokens / request_e2e_s` as Output
TPS. That includes prefill and TTFT and materially understates decode speed.

## Definitions

The benchmark now records a monotonic timestamp for every SSE `data:` event and
for every effective output event. An effective output event is the first delta
containing non-empty `content`, `reasoning_content`, or `tool_calls`.

It reports:

- `request_e2e_s` and `first_event_s`;
- `stream_decode_window_s = request_e2e_s - first_event_s`;
- `output_rate_e2e` and `output_tps_decode`;
- per-request and aggregate ITL P50/P90;
- `prompt_tps_total`, `prompt_tps_uncached`, and `cache_tps`;
- `score_overlap`, where total prompt TPS and cache TPS overlap;
- `score_disjoint`, where Input TPS excludes cached tokens;
- raw `sse_event_times_s` and `output_event_times_s` for recomputation.

Compatibility fields remain available: `ttft_p90_s`, `output_tps_p10`,
`input_tps`, and `weighted_score`. They map to first-event P90, decode TPS P10,
total prompt TPS, and overlap score respectively.

ITL is measured between effective output SSE events. This equals token latency
for the current vLLM one-token-per-event stream; raw timestamps remain present
so future clients can apply a different grouping rule.

## Gates

## Live validation

The existing healthy TP=4 service was used only to validate measurement. No
engine code was restarted or changed.

One uncached request with eight output tokens produced:

```text
success_rate            1.0
first_event_p90_s       3.6863
output_rate_e2e_p10     1.7534
output_tps_decode_p10   9.1300
ITL P50                 0.1241 s
SSE data events         11
effective output events 8
```

The gap between 1.75 e2e tokens/s and 9.13 decode tokens/s confirms that the old
metric mixed prefill into Output TPS. This is a measurement correction, not an
engine performance improvement. Historical scores must not be compared to the
new score without recomputation from request-level evidence.

## Decision

Keep. Use these fields for all subsequent strict A/B experiments. Report both
overlap and disjoint scores until the organizer's exact Input TPS attribution is
confirmed.
