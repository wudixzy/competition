## Interpretation

Routed MoE is the dominant **currently measured** path for both prefill and
decode. This is consistent with the model executing 40 routed MoE blocks per
token. The `T == 1` decode branch already gathers the eight selected experts on
device and uses one flattened W13 `F.linear` plus one W2 `bmm`; it does not call
`.tolist()` or run a Python expert loop. The `bincount(...).tolist()` grouping
path applies only to `T > 1` prefill/multi-token execution.

The current timer covers only routed expert calculation. It excludes the
router projection, shared-expert gate/up/down projections, shared gate,
tensor-parallel all-reduce, residual/norm work, sampler, and other layer-level
overhead. Consequently, the 78.82% figure is a share of instrumented regions,
not proof that routed expert calculation is 78.82% of observed ITL.

Runtime capability evidence in `E_CAP_01_RUNTIME_20260713.md` already rejects
the planned vendor FusedMoE path at the symbol gate: the target ixformer and
`torch.ops._moe_C` entry points are absent. This satisfies the plan's condition
for starting a custom fallback experiment.

The coverage run used intrusive synchronized timers and is only suitable for

## Decision

Do not implement another T=1 selected-expert path blindly. First microbenchmark
the existing path's routing, selected-weight gather, flattened W13, activation,
W2 bmm, and final reduction against supported PyTorch alternatives. In
parallel, evaluate shared-expert projection fusion; the 40 sequential TP
all-reduces cannot be combined across dependent layers and should be treated as
a collective latency floor unless the model-parallel implementation itself can
be improved.

Preserve the existing grouped prefill path as the reference/fallback. Any
candidate still requires exact routed-output parity, full greedy token equality,
at least 1.3x T=1 MoE microbench speedup, three paired decode A/B runs, full
smoke, 99.5K cold/warm equality, and zero non-finite/CUDA/NCCL errors.

The fixed evaluator YAML remains unchanged.
